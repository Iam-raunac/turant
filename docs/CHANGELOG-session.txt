PART 2 — DYNAMIC LEARNING + DYNAMIC PROFILES
==============================================================================
GOAL (your request): stop using hardcoded preferences. When a user orders an
item (e.g. biryani) repeatedly, it should resurface in future relevant carts.
Profiles should be created on the fly, not pre-built.

BACKEND — backend/lambda/parse_and_generate/lambda_function.py
- Added `import time` and `from decimal import Decimal`.
- New function record_order(user_id, items, name):
    * Auto-creates a user profile on first order (no seeding needed).
    * Keeps a running tally: item_counts {product_id: times}, item_names,
      category_counts, order_count, last_order_ts.
    * Preserves any seeded demographic fields if they exist.
- Rewrote build_personalization_block() + new build_learned_history_block():
    * Injects the user's REAL order history into the prompt.
- New _json_safe() helper: converts DynamoDB Decimal values so json.dumps works.
- lambda_handler now ROUTES on an "action" field (all on the SAME endpoint, so
  no new API Gateway routes are needed):
    * action="record_order"  -> records an order into learned history
    * action="get_profile"    -> returns what's been learned (for the UI)
    * default (no action)     -> generate a cart (unchanged behavior)
  NOTE: this requires the Lambda's IAM role to allow dynamodb:PutItem on
  UserPreferences (per HANDOFF it has full DynamoDB access, so you're covered).

CATALOG — backend/data/catalog.json

FRONTEND

PART 3 — FIX: IRRELEVANT ITEMS BEING ADDED TO EVERY CART
==============================================================================
PROBLEM YOU HIT: after ordering guest-dinner items, EVERY later cart got
padded with Naan, Paneer, Coca-Cola, Haldiram — even a Raksha Bandhan rakhi
request and a "remote battery" request. Two root causes:

  (1) The learned-history prompt said "STRONGLY prefer including" past orders,
      so the model forced favorites into unrelated situations.
  (2) The catalog was food/guest heavy and had NO batteries, NO rakhi, etc., so
      for non-food needs the model substituted food instead of admitting no match.

FIXES — backend/lambda/parse_and_generate/lambda_function.py
- Rewrote LEARNED-HISTORY RULES: relevance to the current situation is now an
  ABSOLUTE gate. Past orders are a TIE-BREAKER ONLY. Explicit "do NOT add" lists
  per situation (e.g. festival/pooja -> never naan/paneer/cola; health -> never
  food). Past orders never justify an off-theme item.
- Relaxed the "must pick 4-6 items" rule -> "a tight 3-item cart beats a padded
  6-item one." Added: if the catalog has no good match, return a smaller/honest
  cart or ask a clarifying question instead of substituting unrelated food.

CATALOG — backend/data/catalog.json (now 52 products, 10 balanced categories)
- essentials: AA Batteries, AAA Batteries, USB-C Cable, Extension Board
- monsoon:    Mosquito Repellent Refill, Compact Umbrella
- festival:   Rakhi Set, Soan Papdi, Roli Chawal Tikka Set
- guests:     Disposable Plates

FRONTEND — frontend/src/mock.js
- applyLearning() changed to RE-RANK ONLY: it boosts items already in a cart
  that the user buys often, but NEVER pads the cart with unrelated favorites
  (keeps mock behavior consistent with the new backend philosophy).

Verified: backend compiles; frontend builds clean.

CAVEAT: relevance is enforced via the PROMPT, not hard code. Nova Lite is a
small model, so rare bleed-through is possible. The robust next step (not yet
done) is a code-side category filter that only allows learned items whose
category matches the detected situation.
